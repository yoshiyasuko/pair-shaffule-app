GASへのproductionデプロイを実行する。未コミット変更の検出からgit push→clasp push→clasp deployまでの一連のフローを自動化し、デプロイ漏れや手順ミスを防ぐ。

## 手順

### 1. 現在の状態を把握

以下を並列で実行：
- `git status` — 未コミットの変更を確認
- `clasp deployments` — 既存デプロイメントのIDとメッセージを取得

### 2. 未コミット変更の処理

未コミットの変更がある場合、AskUserQuestionツールで確認する：
- 質問文: 「未コミットの変更があります。コミットしてからデプロイしますか？」
- 選択肢: ["コミットしてデプロイ", "デプロイを中止"]
- 「コミットしてデプロイ」→ Skillツールで `git-workflow:commit` スキルを実行する（引数: "skip-push skip-hooks"）。完了後に続行
- 「デプロイを中止」→ 処理を終了

変更がなければそのまま次へ進む。

### 3. git push

```bash
git push
```
プッシュに失敗した場合（リモート未設定など）は警告を表示するが、デプロイは続行する。GASへのデプロイはgitとは独立しているため。

### 4. clasp push

```bash
clasp push
```
失敗した場合はエラーを表示して処理を終了する。GASにコードが反映されなければデプロイしても意味がないため。

### 5. デプロイメッセージの生成

`git log` で前回デプロイ以降のコミットを確認し、変更内容を簡潔な英語1文で要約する。
例: "Add pair shuffle animation and sound effects"

### 6. デプロイの実行

手順1で取得した既存デプロイメントのIDを使って更新する：
```bash
clasp deploy -i <deploymentId> -d "<デプロイメッセージ>"
```
既存デプロイメントが見つからない場合は新規作成する：
```bash
clasp deploy -d "<デプロイメッセージ>"
```

productionデプロイは既存IDを上書きすることでURLを維持する。新規作成するとURLが変わってしまう点に注意。

### 7. 結果の表示

デプロイ完了後に以下を表示する：
- デプロイメントID
- デプロイメッセージ
- WebアプリURL（`clasp open --webapp` で開けることも案内）
