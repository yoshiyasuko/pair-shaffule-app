あなたはgit commitのエキスパートです。以下の手順に厳密に従ってコミットを作成してください。

## 手順

### 1. 変更状況の確認

以下のコマンドを並列で実行し、現在の変更内容を把握する：
- `git status`（未追跡ファイル・変更ファイルの確認）
- `git diff`（ステージされていない変更の確認）
- `git diff --cached`（ステージ済みの変更の確認）
- `git log --oneline -10`（直近のコミットメッセージのスタイル確認）

### 2. 論理的な単位への分割（必須）

変更内容を分析し、**論理的にまとまった単位ごと**にコミットを分割する。
- 1つのコミットに無関係な変更を混ぜてはならない
- 例：機能追加とバグ修正は別コミット、リファクタリングと新機能は別コミット
- 分割が必要な場合は、ファイル単位または変更単位で `git add` を使い分ける
- `git add -A` や `git add .` は使わない。必ず個別のファイルを指定する

### 3. git-cz形式のコミットメッセージ作成

Conventional Commits (git-cz) 形式でコミットメッセージを作成する：

```
<type>(<scope>): <subject>

<body>
```

**type** は以下から選択：
- `feat`: 新機能
- `fix`: バグ修正
- `docs`: ドキュメントのみの変更
- `style`: コードの意味に影響しない変更（空白、フォーマット、セミコロン等）
- `refactor`: バグ修正でも機能追加でもないコード変更
- `perf`: パフォーマンス改善
- `test`: テストの追加・修正
- `build`: ビルドシステムや外部依存に影響する変更
- `ci`: CI設定ファイル・スクリプトの変更
- `chore`: その他の変更（ソースやテストを変更しない）

**scope** は変更対象の範囲（省略可）。
**subject** は変更内容の簡潔な説明。**必ず英語で記述する**。命令形で、末尾にピリオドを付けない。
**body** は変更の動機や詳細（省略可）。**必ず英語で記述する**。

### 4. コミットの実行

- コミットメッセージは必ずHEREDOCで渡す：
```bash
git commit -m "$(cat <<'EOF'
<type>(<scope>): <subject>

<body>
EOF
)"
```
- 複数コミットが必要な場合は、1つずつ順番にステージ→コミットを繰り返す

### 5. コミット結果の表示

全てのコミットが完了したら、以下を実行してコミット一覧を表示する：
```bash
git log --oneline -10
```

### 6. プッシュの確認

AskUserQuestionツールを使って、リモートにプッシュするかユーザーに尋ねる。
- 質問文: 「リモートにプッシュしますか？」
- 選択肢: ["はい", "いいえ"]
- ユーザーが「はい」と回答した場合のみ `git push` を実行する

## 注意事項

- シークレット情報を含むファイル（.env, credentials等）はコミットしない
- pre-commitフックが失敗した場合は問題を修正して新しいコミットを作成する（--amendしない）
- `--no-verify` や `--no-gpg-sign` は使用しない
- 変更がない場合は空コミットを作成しない
