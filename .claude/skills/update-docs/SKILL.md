コード変更内容を分析し、README.md と CLAUDE.md の更新が必要かを判断・更新する。コード変更後やコミット前に呼び出して、ドキュメントの陳腐化を防ぐ。機能追加・アーキテクチャ変更・設定変更・スキル追加など、ドキュメントに影響する変更があった場合に使用する。

また、変更がなくてもドキュメントの記述が実態と乖離していないか検証する機能も持つ。定期的に呼び出して、ドキュメントの正確性を保つ。

## 手順

### 1. 動作モードの決定

以下を並列で実行して、未コミットの変更があるかを確認する：
- `git diff HEAD` — 未コミットの変更
- `git diff --cached` — ステージ済みの変更
- `git log --oneline -5` — 直近コミット（最近の変更の流れを把握するため）

**変更がある場合** → **差分モード**（ステップ2へ）
**変更がない場合** → **検証モード**（ステップ3へ）

## 差分モード（コード変更に伴うドキュメント更新）

### 2. 差分ベースの更新判断

Read ツールで README.md と CLAUDE.md の内容を読み込み、変更内容とドキュメントの記述を照合して更新が必要かを判断する。

**CLAUDE.md — 開発者向けの技術リファレンス**

更新が必要：
- ファイルの追加・削除・役割変更（アーキテクチャセクション）
- サーバー/クライアントの関数・APIの追加・変更・削除
- デプロイ手順や設定の変更
- 設計上の重要な意思決定
- スキル（スラッシュコマンド）の追加・変更・削除

**README.md — ユーザー向けの利用ガイド**

更新が必要：
- ユーザーに見える機能の追加・削除
- セットアップ手順の変更
- 使い方の変更
- データ形式（スプレッドシート等）の変更

**更新不要と判断してよいケース：**
- UIのスタイル調整のみ（色・フォント・アニメーション）
- 動作が変わらないバグ修正
- 外部インターフェースに影響しない内部リファクタリング
- コメント・ドキュメント内の誤字修正のみ

迷ったら「不要」寄りに判断する。ドキュメントの過剰な更新はノイズになる。

→ ステップ5（更新案の提示と確認）へ

## 検証モード（ドキュメントと実態の整合性チェック）

### 3. ドキュメントとソースの読み込み

以下を並列で読み込む：
- README.md
- CLAUDE.md
- ドキュメントに記載されているソースファイル（アーキテクチャセクション等に列挙されたファイル）
- `.github/workflows/` 配下のワークフローファイル
- `.claude/skills/` 配下のスキル定義ファイル一覧
- `appsscript.json`、`.clasp.json` などの設定ファイル

### 4. 整合性チェック

ドキュメントの記述を、実際のファイル内容と突き合わせる。以下のチェックリストに沿って不整合を探す。

**CLAUDE.md のチェック項目：**

| チェック対象 | 照合先 | 確認内容 |
|---|---|---|
| アーキテクチャのファイル一覧 | 実際の `src/` ディレクトリ構成 | 記載ファイルが存在するか、未記載のファイルがないか |
| 関数名・API名 | 各ソースファイル | 記載の関数が実際に存在するか、シグネチャが一致するか |
| コードスニペット・パス文字列 | 各ソースファイル | `include()` のパス等、ドキュメント内のコード片が実態と一致するか |
| 設定値の記述 | `appsscript.json`、`.clasp.json` | タイムゾーン・ランタイム・スコープ等の記述が実際の設定と一致するか |
| CI/CDセクション | `.github/workflows/*.yml` | トリガー条件・ステップ・必要なSecrets名が実態と一致するか |
| スキル一覧テーブル | `.claude/skills/` の実ファイル | 記載コマンドと実際のファイルが一致するか、概要が適切か |
| スキル連携フロー | 各スキル定義ファイル | フロー図が実際のスキル間呼び出しと一致するか |

**README.md のチェック項目：**

| チェック対象 | 照合先 | 確認内容 |
|---|---|---|
| 機能一覧 | クライアント/サーバーのソースコード | 記載機能が実装されているか、未記載の主要機能がないか |
| セットアップ手順 | 設定ファイル・ソースコード | 手順通りに進めて正しく動作する記述か |
| スプレッドシート形式 | `Code.js` の読み取りロジック | 列の用途・ヘッダー行の扱いが実装と一致するか |
| デプロイ手順 | ワークフローファイル・`.clasp.json` | 手動/自動デプロイの記述が実態と一致するか |

不整合が見つからない場合は「README.md / CLAUDE.md の記述は実態と整合しています」と伝えて終了。

→ ステップ5（更新案の提示と確認）へ

## 共通ステップ

### 5. 更新案の提示と確認

更新が必要と判断した場合、AskUserQuestion ツールで更新内容を提示し確認を取る：

- 質問文に、どのファイルのどの箇所をどう変更するかを具体的に記載する
- 選択肢: ["更新する", "スキップ"]
- README.md と CLAUDE.md の両方に更新がある場合でも、1つの質問にまとめてよい

更新が不要と判断した場合は理由を伝えて終了。

### 6. 更新の実行

ユーザーが「更新する」を選んだ場合のみ、Edit ツールで該当箇所を更新する。

- 既存のスタイル・トーン・構造を維持する
- 変更に関係する箇所だけを編集する（全体の書き換えはしない）
- 推測で書かない。コードから読み取れる事実のみを記述する

### 7. 結果の報告

更新した場合は、変更箇所を簡潔に報告する。`/commit` から呼ばれている場合は報告後に制御を返す。
